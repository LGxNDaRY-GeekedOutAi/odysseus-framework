Security Architecture

Security Is Not a Feature — It's the Foundation

Odysseus treats security as a first-class runtime citizen, not an afterthought. Every dangerous operation passes through multiple independent gates before execution.

🔒 Layer 1 — CUDA Enforcement (Hard Block)

CPU-only operation is permanently disabled. This is not a configuration option. The system will not start, will not load a model, and will not run inference without a CUDA-compiled llama-cpp-python wheel. This prevents silent RAM-only model loading which would expose every user query to swap disk access.

Hard requirement: llama-cpp-python must be compiled with CMAKE_ARGS='-DGGML_CUDA=ON'. Fix: $env:CMAKE_ARGS='-DGGML_CUDA=ON'; pip install llama-cpp-python --force-reinstall --no-cache-dir

🔑 Layer 2 — USB Physical Key (Production Patch Gate)

Applying any code patch to production requires a physical USB key to be inserted. The USB lock state is checked server-side via /self-workspace/usb-unlock before any file copy to production is permitted. This prevents unauthorized remote code modification even by authenticated master-profile users.

🏰 Layer 4 — Workspace Sandbox

OpenClaw's file tools (read_file, write_file, list_files) enforce a workspace root boundary via _resolve_workspace_path(). All path arguments are canonicalized and checked against LORD_OF_AGENTS_WORKSPACE_ROOT — any traversal attempt outside the workspace root returns a PermissionError.

⚖️ Layer 5 — Reverse Engineering Authorization Policy

The REVERSE_ENGINEERING workflow mode enforces a multi-step authorization gate before any analysis begins. Scope declarations, legal authorization metadata, and policy checks must all pass before the Lab Bench stages execute. The Critic agent further validates that outputs remain within defensive/remediation scope.

The Agent Roster

Every Intelligence in the System

Odysseus is not a single model. It is an ecosystem of purpose-built agents — each with a defined role, toolset, and scope — orchestrated by a central intelligence layer.

⬡ Odysseus — The Core Orchestrator

The primary persona. When no specialist workflow is active, Odysseus acts as the user's collaborative AI partner and orchestrator. It has an inner monologue (streamed via <thinking> tags), proposes plans before acting, requests confirmation before destructive operations, and summarizes every action in natural language. It is the face the user sees, and the mind that routes work to every other agent.

Variants: odysseus_software_dev · odysseus_game_designer · odysseus_music_composer · odysseus_reverse_engineer · odysseus_research_rd — each tuned to their domain but sharing the same core identity and communication style.

⚡ Hermes — The Task Dispatcher

Hermes is the central orchestrator of the agent squad — a more structured, protocol-driven dispatcher than Odysseus. Its primary function is to receive high-level goals and decompose them into a precise, sequential action plan before execution. It operates via a strict ANALYZE → PLAN → EXECUTE → OBSERVE → CONCLUDE loop, making it ideal for complex multi-step operations where sequencing accuracy is critical. It has a .hermes local artifacts folder for desktop-level operations.

Role: Bridge between human intent and machine execution. Sub-agent delegation hub.

🦞 OpenClaw Registry — The Tool Layer

OpenClaw (openclaw_registry.py) is the universal tool registry for the entire system. Every capability an agent can call — from running a shell command to searching the web to invoking a sub-agent — is registered here as a BaseTool implementation. The registry enforces workspace path sandboxing on all file operations and manages the active brain/memory context injected by AgentCore before each run.

Registered Tools

command read_file write_file list_files web_search fetch_url researcher_agent coder_agent critic_agent semantic_search pinokio_pipeline save_fact echo thought
CommandTool requires both ODYSSEUS_ENABLE_COMMAND_TOOL=1 AND ODYSSEUS_COMMAND_TOOL_MAINTENANCE_WINDOW=1. Dual-flag prevents accidental command execution in production.

🎨 Pinokio — The Media Generation Engine

Pinokio is an external multi-modal AI generation service running on the same machine. Odysseus communicates with it via a local HTTP API, dispatching asset jobs through the pinokio_pipeline OpenClaw tool. The PinokioAgent class implements robust endpoint discovery — probing /chat, /query, and /execute endpoints in order until one responds.

Before every Pinokio job dispatch, the VRAM Governor pre-classifies the asset type, transitions the GGUF model to the appropriate state (PARTIAL_GPU or RAM_CACHE), dispatches the job, then restores the model from warm cache when the job completes.

RAM Warm Cache

When the model is ejected, all parameters (model_path, n_ctx, n_gpu_layers, n_threads, temperature) are stored in WarmModelCache. The GGUF weights remain in OS page cache in your 64 GB DDR5. Reload via load_local_model() reads from page cache rather than NVMe — cutting reload time from 30–60 s down to ~3 s.
